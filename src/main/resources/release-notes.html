v4.0.27

2026-03-20

New Features

  • QC equipment instruction messages corrected for DSCH — equipmentInstructionType now sends LIFT/PLACE, proper equipmentInstructionText, empty destinationNodeId/destinationNodeName for QC, moveKind from WI, pinning-based instructionDetails (SKIP_PINNING/GO_PINNING)
  • New WI fields: isoType, freightKind, pinning — wired through Kafka consumer, DemoServer, JsonSerializer, schedule-viewer form, and work instructions page
  • Container fields populated in equipment instructions: containerId, fromPosition, isoType, freightKind, containerTruckPosition (from WI jobPosition)
  • targetChe field on actions — non-TT actions (QC, RTG) receive the assigned TT name when a truck is allocated for the same containerIndex
  • TT actions processed first in activation pass — ensures truck allocation and targetChe propagation happen before QC/RTG actions activate
  • Truck assignments preserved across reschedules — transferTruckAssignments() carries cheShortName and targetChe from old schedule to new
  • Completion conditions framework — actions can have static CompletionCondition records that must be satisfied by external events before the action auto-completes
  • CompletionConditionEvaluator interface with QCAssetEventEvaluator — QC_LIFT completes on “Lifted container from vessel” asset event, QC_PLACE on “Placed container on truck”
  • Terminal Simulator (com.wonderingwizard.simulator) — embedded in same image, started with RUN_MODE=simulator. Consumes QC equipment instructions, responds with asset events and “QC Discharged Container” WI events
  • Simulator tracks WI state by consuming the WI topic (Avro) to construct discharge events with full WI fields
  • DemoEventProducer abstraction — when Kafka is enabled, WI events from the UI are published to Kafka (Avro) instead of processed directly, enabling realistic round-trip through the simulator

Improvements

  • destinationCHEShortName on QC_PLACE now correctly set to the assigned TT (not the RTG)
  • carryCHEShortName on QC instructions set to the assigned TT
  • Play clock uses fire-and-forget event submission — prevents HTTP request timeouts when clock is ticking
  • Settings file path trimmed to handle trailing whitespace from IDE env vars

New Features

  • Navigation sidebar with live badges:
    • Work Instructions — count of active WIs (not Complete or Cancelled)

  • Manual time jumps (+1s, +1m, +5m, +15m) now also advance the play clock’s last-sent time so the clock continues seamlessly from the new position

Improvements

  • Removed resolution (s/tick) control from schedule viewer — backend always advances 1 second per tick
  • FES System Time label added to the clock display
  • clock.autostart setting (default true) — controls whether the play clock starts automatically on server startup

v4.0.24

2026-03-19

New Features

  • Equipment location occupancy tracking — computes QC and RTG standby/under occupancy from active schedule state in real-time
  • Location-based action conditions with two modes:
    • BLOCK (ACTIVATE mode) — prevents TT from entering a position that is already occupied (e.g., don’t drive to QC standby if standby is occupied)
    • SKIP mode — auto-completes TT approach actions when the next position is free, allowing the truck to skip ahead (e.g., skip QC pull if standby is free)
  • QC skip conditions apply to all load modes; RTG skip conditions apply to DSCH only
  • Location conditions visible in schedule viewer as LOCATION_BLOCK and LOCATION_SKIP conditions on pending TT actions, with override support
  • Equipment Locations page (/locations) — live view of QC and RTG standby/under positions showing truck name, work queue, work instruction, and container ID
  • LOCATION_SKIPPED completion reason for auto-completed approach actions

Bug Fixes

  • Fixed SSE reconnect loop — new connections now send initial state only to the new client (not broadcast to all) on a virtual thread to avoid blocking the HTTP handler

v4.0.22

2026-03-19

Performance

  • JSON serialization moved off the event loop — buildState() (fast snapshot) runs on the event loop, serialize() (slow string building) runs on the broadcast thread
  • Steps list uses List.copyOf() for thread-safe off-loop serialization
  • Adaptive SSE broadcast debounce — 50ms when idle (responsive UI), scales up to 500ms under queue pressure (50 + queueSize × 50, clamped)

v4.0.4

2026-03-18

Improvements

  • ActionStatus moved from ScheduleRunnerProcessor inner enum to a domain-level enum on the Action record — action status is now a first-class field (action.status()) instead of tracked externally via separate sets
  • Canceled containers no longer trigger spurious reschedules — when a container's actions are force-completed (e.g., TT unavailable), its QC Discharged event is ignored and the non-canceled twin's discharge is correctly recognized as the expected next WI

Internal

  • Removed activeActionIds and completedActionIds sets from ScheduleState; all status tracking now lives on the Action record via withStatus()
  • New WorkInstructionCanceled side effect (propagating event) informs WorkQueueProcessor when work instructions are canceled, preventing false order-mismatch reschedules
