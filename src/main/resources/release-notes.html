v4.0.15

2026-03-18

Bug Fixes

  • Fixed Prometheus metric name mismatch — removed .setUnit("ms") from OTEL histograms to prevent the exporter from appending _milliseconds suffix (e.g. fes_kafka_processing_duration_ms_milliseconds_bucket → fes_kafka_processing_duration_ms_bucket)
  • Added warmup metric recording at startup so Prometheus always gets samples even before the first Kafka message arrives

v4.0.14

2026-03-18

Improvements

  • Metrics now record in milliseconds with custom histogram buckets (0.01ms to 5000ms) — shows actual per-event processing time instead of coarse second-level buckets
  • Metric names changed to fes_kafka_processing_duration_ms and fes_engine_processing_duration_ms
  • Grafana dashboard updated to match new metric names and display ms units

v4.0.13

2026-03-18

New Features

  • OpenTelemetry metrics with Prometheus exporter on port 9464 — fes_kafka_messages_total (counter by topic), fes_kafka_processing_duration_seconds (histogram by topic), fes_engine_processing_duration_seconds (histogram by event type)
  • Grafana dashboard (grafana/fes-dashboard.json) with 12 panels: throughput, p50/p90/p99 per topic and per event type
  • POST /api/che-logical-position endpoint for sending position events via HTTP
  • E2E performance test that publishes 10k Avro CheLogicalPosition messages to Kafka and verifies processing within 20s

v4.0.12

2026-03-18

New Features

  • Kafka consumer for CheLogicalPosition topic — receives truck GPS positions and map node info
  • Trucks table now shows position columns: Node ID, Node Name, Latitude, Longitude, and HDOP

Details

  • TTStateProcessor handles CheLogicalPositionEvent alongside ContainerHandlingEquipmentEvent — position data is only stored for trucks that already exist (no phantom entries)
  • Position data is included in the /api/state response as truckPositions and serialized by JsonSerializer

v4.0.11

2026-03-18

New Features

  • Path highlighting in pathfinder — found routes are drawn as a thick orange line on the digital map with green start and red end markers
  • 40ft container standby locations — alt_name and apmt_poi_standby_bay_40 tags are now parsed; pathfinder has a 20ft/40ft radio toggle for standby lookup

Improvements

  • Priority-weighted pathfinding — Dijkstra now uses driveTime × priority as cost, preferring low-priority (main) roads over high-priority service lanes, matching the C# WeightCalculator
  • Standby location validation — warns at map parse time if a standby references a non-existent POI
  • Corrupted OSM way filtering — ways from truncated gzip tails are detected and skipped, eliminating phantom diagonal lines on the map

v4.0.7

v4.0.6

  • Clickable ⚠ icon on force-completed actions now shows a popup with the completion reason

v4.0.5

2026-03-18

Bug Fixes

  • Fixed TT unavailable boundary — trucks are now recoverable (reset + reallocate) through TT_DRIVE_UNDER_QC; cancellation only triggers once TT_HANDOVER_FROM_QC is active or completed

v4.0.4

2026-03-18

Improvements

  • ActionStatus moved from ScheduleRunnerProcessor inner enum to a domain-level enum on the Action record — action status is now a first-class field (action.status()) instead of tracked externally via separate sets
  • Canceled containers no longer trigger spurious reschedules — when a container's actions are force-completed (e.g., TT unavailable), its QC Discharged event is ignored and the non-canceled twin's discharge is correctly recognized as the expected next WI

Internal

  • Removed activeActionIds and completedActionIds sets from ScheduleState; all status tracking now lives on the Action record via withStatus()
  • New WorkInstructionCanceled side effect (propagating event) informs WorkQueueProcessor when work instructions are canceled, preventing false order-mismatch reschedules

v4.0.3

2026-03-18

New Features

  • TT unavailable event handling — when a terminal truck becomes unavailable mid-operation, the system either reassigns work or force-completes remaining actions depending on workflow progress
  • Sub-processor pattern (ScheduleSubProcessor) for encapsulating schedule event handling logic, with controlled state access via ScheduleContext
  • TruckUnassigned side effect emitted when a truck is removed from actions
  • CompletionReason enum and completionReason field on actions and ActionCompleted side effect to track force-completion causes

Improvements

  • Schedule viewer shows force-completed actions with gray background and ⚠ icon instead of the normal green check mark
  • Before TT-under-QC activation: truck is unassigned from all TT actions and a new truck is allocated if available
  • After TT-under-QC activation: all remaining actions for the container and its twin are completed with reason TT_UNAVAILABLE
